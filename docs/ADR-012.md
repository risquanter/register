# ADR-012: Service Mesh Strategy

**Status:** Accepted (awaiting implementation)  
**Date:** 2026-01-17  
**Tags:** infrastructure, service-mesh, resilience, authentication, authorization

---

## Context

- **Resilience** (retries, circuit breaking, timeouts) should be handled at infrastructure level, not application code
- **Authentication/Authorization** should be externalized to dedicated services (Keycloak, OPA)
- Service mesh provides **uniform policy enforcement** across all services
- Application code should focus on **business logic**, not infrastructure concerns
- Development environment can **let failures happen** (no mesh required locally)

---

## Decision

### 1. Istio Ambient Mode as Service Mesh

Istio Ambient Mode is the chosen service mesh implementation:

- **ztunnel** — L4 proxy for mTLS between all pods (zero-config, low overhead)
- **Waypoint proxies** — L7 features (JWT validation, routing, authorization) deployed per-namespace

**Rationale:**
- Native JWT/OIDC validation via `RequestAuthentication`
- First-class OPA integration via `ext_authz` filter
- Lower resource usage than sidecar mode
- CNCF Graduated project with strong ecosystem

### 2. Keycloak for Identity (OAuth2/OIDC)

Keycloak provides:
- User authentication (login flows)
- JWT token issuance
- Role management and JWT claims mapping
- JWKS endpoint for mesh JWT validation

### 3. OPA for Authorization

OPA operates at the **mesh layer** (Istio `ext_authz`), before the request reaches the application. It answers a purely claim-based question: "does this role claim permit attempting this operation type?"

- Input: JWT claims + HTTP method/path only — **no relationship data, no data bundles**
- Speed: sub-millisecond (sidecar/in-process evaluation)
- Policy authors: security team (Rego) — changes deployable without an app release
- Defense-in-depth: security team can push emergency overrides (deny writes during incident, require audit headers) independently of engineering

OPA and SpiceDB use **AND logic**: both must allow for the request to proceed. OPA deny = 403 immediately (SpiceDB never queried). OPA allow + SpiceDB deny = 403. Neither can unilaterally grant access past the other.

See [AUTHORIZATION-PLAN.md Task L2.4](./AUTHORIZATION-PLAN.md) for the full responsibility split table and request flow diagram.

### 4. No Application-Level Resilience Code

The following are **delegated to Istio**, not implemented in Scala:
- Retries (via `VirtualService`)
- Circuit breaking (via `DestinationRule.outlierDetection`)
- Timeouts (via `VirtualService`)
- Rate limiting (via EnvoyFilter or waypoint config)

### 5. Minimal Service Code for Auth

Service code handles only:
- Claim header reading (mesh validates JWT and injects pre-parsed claims as plain HTTP headers; app reads `x-user-id` — zero JWT code in the application)
- Authorization checks — `check()` call to SpiceDB ("can user X do Y on resource Z?")
- Audit logging (structured logging of actions)

**The application is a pure Policy Enforcement Point (PEP).** It never writes authorization data (`grant()`/`revoke()`/tuple writes). Writing authorization data is the exclusive role of the Policy Administration Point (PAP): the CI/CD provisioning job (`AuthzProvisioning`) and ops-path tooling (`zed` CLI). See [ADR-024](./ADR-024-externalized-authorization-pep-pattern.md) for the full architectural principle and code smells.

**Estimated:** ~100-200 lines across entire backend.

### 6. Claim Header Injection

`outputClaimToHeaders` injects validated JWT claims as plain headers after waypoint processing — but Istio does **not** strip incoming `x-user-*` headers from external requests by default. Without explicit stripping, an external client can forge `x-user-id: <uuid>` and bypass JWT validation on any route that lacks waypoint JWT processing. See [❌ Trusting External Claim Headers Without Stripping](#trusting-external-claim-headers-without-stripping) for the attack pattern and correct configuration.

- **Strip at ingress (non-optional):** waypoint or `EnvoyFilter` must remove `x-user-id`, `x-user-email`, `x-user-roles` from all external requests before claim injection runs
- **ztunnel is insufficient:** ztunnel mTLS secures pod-to-pod traffic inside the mesh; the external ingress path is not covered
- **Post-strip trust model:** `x-user-id` present = infrastructure assertion (authenticated Keycloak user); absent = unauthenticated
- **Acceptance criterion:** staging must confirm a forged `x-user-id` header with no valid JWT is stripped — not forwarded to the backend

## Code Smells

### ❌ Application-Level Circuit Breaker

```scala
// BAD: Resilience in application code
val circuitBreaker = CircuitBreaker.make(...)
circuitBreaker.protect(irminClient.query(...))
```

```scala
// GOOD: Let Istio handle it
irminClient.query(...)  // Mesh provides retries/circuit breaking
```

### ❌ Manual JWT Validation

```scala
// BAD: Validating JWT in application
def validateJwt(token: String): IO[AuthError, Claims] =
  for {
    key   <- fetchJwksKey(...)
    claims <- verifySignature(token, key)
    _     <- checkExpiry(claims)
  } yield claims
```

```scala
// GOOD: Trust mesh validation, read pre-validated claim headers — zero JWT code in the app
def extractUserContext(headers: Headers): IO[AuthError, UserContext] =
  // Mesh validated JWT and injected x-user-id; app reads the plain header value
  headers.get("x-user-id") match
    case Some(sub) => UserId.fromString(sub).bimap(e => AuthForbidden(e.mkString), UserContext(_))
    case None      => ZIO.fail(AuthForbidden("Missing x-user-id header — unauthenticated request or mesh bypass"))
```

### ❌ Syncing Relationship Data Into OPA Bundle

```rego
# BAD: OPA reads relationship data from a synced bundle
allow {
    data.workspaces[input.workspace_id].members[_] == input.user
}
```

```rego
# GOOD: OPA reads JWT claims only — no bundle data
allow if {
    some role in input.jwt.realm_access.roles
    role in {"analyst", "editor", "team_admin"}
}
# Relationship checks ("is this user a member of this workspace?") are SpiceDB's job
```

### ❌ Trusting External Claim Headers Without Stripping

```yaml
# BAD: Istio injects x-user-id AFTER JWT validation — but does NOT strip the header from
# external requests by default. An attacker can send `x-user-id: <victim-uuid>` with no JWT
# and the app receives the forged value as if it were mesh-injected.
#
# This bypasses JWT validation entirely on any path that does not go through the waypoint
# JWT check (public routes, misconfigured AuthorizationPolicy exceptions).
```

```yaml
# GOOD: Configure the waypoint to strip x-user-id, x-user-email, x-user-roles
# from incoming external requests BEFORE JWT validation and claim injection run.
# The exact mechanism (EnvoyFilter, waypoint header policy) depends on Istio ambient
# mode configuration — but stripping is a required invariant, not optional hardening.
#
# Trust model after stripping + JWT validation + outputClaimToHeaders injection:
#   - x-user-id present  → infrastructure assertion: authenticated Keycloak user
#   - x-user-id absent   → unauthenticated; modes requiring identity fail closed
```

## Development Environment

**Decision:** Let it fail (no service mesh locally)

- Docker Compose runs services directly (no sidecars)
- No JWT validation locally (trust all requests or use mock)
- Failures are acceptable during development
- Integration testing with mesh happens in staging/CI

---

## Implementation Guidance

For `outputClaimToHeaders` configuration and UserContext extraction from mesh-injected claim headers, consult Istio `RequestAuthentication` documentation and the Keycloak JWT claim mapping reference directly.

**Before implementing:**
1. Review Keycloak OIDC and Istio JWT forwarding documentation
2. Validate applicability against current codebase state
3. Adapt patterns to project conventions (ADR-010, ADR-011)

**Note:** The Cheleb reference architecture is relevant for ZIO + PostgreSQL layer patterns (see Tier 1.5), but does **not** contain Keycloak or OAuth2 integration patterns.

Do **not** reinvent established patterns—leverage proven implementations first.

---

## Consequences

**Positive:**
- Consistent resilience across all services (mesh-enforced)
- Centralized auth policy (Keycloak + OPA)
- Reduced application code complexity
- Zero-trust security model (mTLS everywhere)

**Negative:**
- Mesh adds operational complexity
- Debugging requires mesh observability tools
- Local dev differs from production (no mesh)

**Neutral:**
- Team must learn Istio concepts
- Staging environment must mirror production mesh config
